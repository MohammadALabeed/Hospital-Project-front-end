## 1. المقدمة ونطاق النظام (Project Scope)
يقدم هذا المشروع الواجهات الأمامية لنظام إدارة مستشفى متطور ومكتمل معزز بلوحات تحكم تفاعلية لإدارة المرضى والمواعيد والسجلات الطبية. تم بناء الواجهة معتمدة على فلسفة **المكونات المعزولة (Component-Based Architecture)** لضمان مرونة الصيانة وسرعة التطوير والاستجابة.

---

## 2. المعمارية التقنية للمشروع (Technical Stack)

|    التقنية     |    الدور الوظيفي في المشروع     |                                                  الأهمية البرمجية                                                  |
| :---           | :---                            | :---                                                                                                               |
| **React 18**   | المكتبة الأساسية لبناء الواجهات | تتيح ميزة الـ Virtual DOM لتحديث العناصر بسرعة فائقة بدون إعادة تحميل الصفحة بالكامل.                              |

## 3. التحليل البرمجي للمكونات (Components Analysis)

بناءً على الهيكلية الهندسية للمجلد الفعلي `/src/components` (الموضح في **Screenshot 2026-06-04 053733.png**)، تم توزيع العمل الوظيفي كما يلي:

### أ. المكونات الرسومية الأساسية (Core Views):
* **`Overview.tsx`:** يمثل اللوحة الترحيبية والإحصائية العامة (عدد المرضى اليوم، المواعيد القادمة، التنبيهات السريعة).
* **`Patients.tsx`:** جدول تفاعلي ذكي لعرض كافة المرضى، مزود بخيارات البحث والفرز المتقدمة.
* **`Appointments.tsx`:** شاشة مخصصة لجدولة المواعيد وعرض الأوقات المتاحة للأطباء وحالات الحجز (مؤكد/ملغي/قيد الانتظار).
* **`MedicalRecords.tsx`:** واجهة تصفح التاريخ الطبي للمريض والتي تعرض التقارير، التشخيصات، والوصفات الطبية السابقة.
* **`Login.tsx`:** صفحة مستقلة آمنة ومحمية للتحقق من هوية المستخدمين وصلاحيات دخولهم للنظام.

### ب. المكونات المساعدة والتخطيط (Layout & Modals):
* **`Layout.tsx`:** الحاضن الرئيسي للموقع (Shell Layout) ويحتوي على شريط التنقل الجانبي والثابت (Sidebar) وشريط الأدوات العلوي لضمان تجربة تصفح موحدة وثابتة.
* **`PatientModal.tsx`:** نافذة منبثقة منبثقة تفاعلية (Modal Component) تُستدعى بشكل مرن لإضافة مريض جديد أو تعديل بيانات مريض حالي دون الحاجة للانتقال لصفحة جديدة، مما يحسن الـ UX.

## 4. طبقة البيانات وفصل المهام (Data Layer & Decoupling)

تم تصميم النظام هندسياً ليفصل تماماً بين التصاميم ومصادر البيانات لاتباع معايير الـ Clean Code:

1. **ملف العقود والأنواع (`types.ts`):** يحتوي على الواجهات (Interfaces) التي تحدد بدقة شكل وحقول البيانات (مثل حقول المريض: المعرف، الاسم، رقم الهاتف، فصيلة الدم). هذا الملف يضمن التزام كافة المكونات بنفس الهيكل البرمجي.
2. **ملف محرك البيانات (`api.ts`):** يمثل **طبقة الخدمات (Service Layer)**. تم تزويد هذا الملف حالياً ببيانات محاكاة برمجية حقيقية (Mock Data) عالية الدقة ليعمل النظام كلياً بشكل مستقل عن الخادم (Serverless State).
3. **خطة التكامل والربط:** ملف `api.ts` مهيأ لاستبدال دوال المحاكاة بدوال اتصال حقيقية (مثل طلبات Axios لـ Restful APIs) بمجرد تشغيل سيرفر الباك إند، دون الحاجة لتعديل سطر كود واحد في ملفات المكونات أو واجهات المستخدم الرسومية.
